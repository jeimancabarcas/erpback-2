## 2. How Manual Sales Are Identified

**Column:** `is_electronic` (boolean, `@Column({ name: 'is_electronic', default: true })`)

**Entity** (`src/modules/sales/entities/invoice.entity.ts`):
```typescript

```

Pattern: `isElectronic === false` means manual. All existing (pre-migration) invoices are `true` via `DEFAULT TRUE`.

## 3. Current Credit/Debit Note Flow

### Electronic invoices (isElectronic === true)
1. `POST /sales/invoices/:id/credit-note` ‚Üí `SalesController.createCreditNote()`
2. `SalesService.createCreditNote()` loads invoice
3. Validates items against invoice items (quantity checks)
4. Builds `factusPayload` with customer, items, taxes
5. Calls `factusGateway.createCreditNote(factusPayload)` ‚Üí DIAN via Factus
6. Persists CreditNote with Factus response data (number, cude, qrUrl, publicUrl)
7. If `correctionConceptCode === '2'` (total annulment), sets invoice status to `CANCELLED`

Same flow for debit notes via `createDebitNote()` ‚Üí `factusGateway.createDebitNote()`.

### Manual invoices (isElectronic === false)
- **Immediately blocked** at the top of both methods:
```typescript
if (!invoice.isElectronic) {
  throw new BadRequestException(
    'No se pueden crear notas de cr√©dito para facturas manuales',
  );
}
```
- No Factus call happens ‚Äî the error is thrown before any processing.

**Net effect:** Manual invoices can never have credit/debit notes. The spec explicitly says this is "Out of Scope."

## 4. How Factus/DIAN PDF Download Is Triggered

### Invoice PDF ‚Äî `downloadInvoicePdf()`
```
GET /sales/invoices/:id/pdf
  ‚Üí SalesController.downloadPdf()
    ‚Üí SalesService.downloadInvoicePdf(id)
      ‚Üí factusGateway.downloadInvoicePdf(invoiceNumber)
        ‚Üí GET {FACTUS_API_URL}/v2/bills/{number}/download-pdf
```

**Critical:** There is **NO guard** for manual invoices. If someone calls this on a `MAN-xxxx` invoice, it will:
1. Call Factus with `MAN-00000001` as the number
2. Factus will return 404 or error (no DIAN record for that number)
3. The error propagates as a thrown exception
4. **No fallback** ‚Äî the user gets a 500 error

### Adjustment Note PDF ‚Äî `downloadAdjustmentNotePdf()`
```
GET /sales/credit-notes/:id/pdf or /sales/debit-notes/:id/pdf
  ‚Üí SalesController.downloadCreditNotePdf() / downloadDebitNotePdf()
    ‚Üí SalesService.downloadAdjustmentNotePdf(id, type)
      ‚îú‚îÄ If noteNumber exists and is NOT a mock (NC-PEND-* / ND-PEND-*):
      ‚îÇ    ‚Üí factusGateway.downloadAdjustmentNotePdf(noteNumber)
      ‚îÇ      ‚Üí GET {FACTUS_API_URL}/v2/adjustment-notes/{number}/download-pdf
      ‚îÇ    On failure: falls through to simulated PDF
      ‚îî‚îÄ Else (mock number or missing):
           ‚Üí generateSimulatedNotePdfBase64()  ‚Üê raw PDF generation
```

This method already has a **fallback** ‚Äî if Factus fails OR if the note has a mock number, it generates a simulated PDF. However:
- The simulated PDF is generated by manually constructing PDF bytes (no library)
- It only shows the note itself, not the full invoice history
- For manual invoices, this method is never reached because `createCreditNote/DebitNote` throws

## 5. Existing PDF Generation Infrastructure

### What exists
- `SalesService.generateSimulatedNotePdfBase64()` ‚Äî a private method that builds a raw PDF via string concatenation:
  - Uses basic PDF operators (BT/ET, Tf, Td, Tj)
  - Single font (Helvetica), single page (612x792), monochrome
  - No tables, no graphics, no proper formatting
  - Content: doc type header, identifier, date, status, CUDE, reference invoice, customer info, concept, amounts, and a disclaimer line
  - Returns base64-encoded string

### What does NOT exist
- **No PDF library** in `package.json` (no pdfkit, pdfmake, jspdf, pdf-lib)
- No centralized PDF service/module
- No invoice PDF generation (only adjustment note simulation)
- No table/column layout code
- No mechanism to combine invoice + notes into a single document

### Dependencies
The `package.json` has `"dependencies"` with NestJS, TypeORM, bcrypt, passport, class-validator, class-transformer, pg, rxjs ‚Äî but **zero PDF libraries**.

## 7. Risks and Edge Cases

### CRITICAL

1. **`downloadInvoicePdf` has NO guard for manual invoices** ‚Äî If a user clicks "Download PDF" on a manual invoice, it calls Factus with `MAN-xxxx` and fails with an unhandled error. This is a production bug in the current code.

2. **No PDF library exists** ‚Äî Any PDF generation (even a simple one) either needs to use raw PDF byte construction (fragile, hard to maintain) or add a dependency like `pdfkit` or `pdf-lib`.

3. **Credit/debit notes are COMPLETELY blocked for manual invoices** ‚Äî The current design prevents any note creation. The user's goal is to UNBLOCK this, but notes must skip Factus and generate internal PDFs instead.

### WARNING

4. **The simulated PDF for adjustment notes is NOT the same as the desired output** ‚Äî The user wants a "single internal PDF with the full history of what was applied to that invoice." The current `generateSimulatedNotePdfBase64()` only shows the note itself, not the full invoice + all notes history.

5. **`findAllNotes()` returns ALL notes regardless of invoice type** ‚Äî If notes are unblocked for manual invoices, the `findAllNotes()` method will show them. This may be desired, but worth noting.

6. **No `isElectronic` filter on `QueryInvoicesDto`** ‚Äî The query endpoint cannot filter by manual/electronic. This was deliberately excluded from the previous change.

7. **`InvoiceItem` stores `purchasePrice` (FIFO cost) which is useful for PDF** ‚Äî Good news: the entity already has cost data that could be shown in a detailed PDF.

8. **`CreditNote` and `DebitNote` store minimal data** ‚Äî They have `referenceCode`, `noteNumber`, `cude`, `correctionConceptCode`, `amount`, `observation`, but **no items array**. The items are not persisted as related entities ‚Äî they're sent to Factus but not stored locally. This means regenerating a detailed PDF showing which items were credited requires reprocessing from the invoice items.

## 8. Suggested Approach

Based on the exploration, here's the recommended approach:

### Phase 1: Fix the Invoice PDF download for manual invoices
- Add a guard in `downloadInvoicePdf()` to check `invoice.isElectronic`
- If manual: generate a PDF locally (not via Factus)
- If electronic: keep current behavior (Factus download)
- This fixes the existing bug

### Phase 2: Unblock credit/debit notes for manual invoices
- Remove (or conditionally skip) the `BadRequestException` guard
- Instead of calling Factus, process the note locally:
  - Generate local note number (`NC-{invoice.invoiceNumber}-{seq}` or similar)
  - Persist the note with appropriate fields (noteNumber, but no cude/qrUrl/publicUrl)
  - Do NOT set `correctionConceptCode === '2'` logic (total cancellation) ‚Äî or decide if it should only affect local status
  - Store credited items details

### Phase 3: Build a proper PDF generation capability
- Add a lightweight PDF library (recommend `pdfkit` or `pdf-lib` ‚Äî no heavy dependencies)
- Create a dedicated `PdfService` or extend `SalesService` with a private method
- Generate a PDF that includes:
  - Invoice header (number, date, customer)
  - Original items table
  - All credit notes applied (with dates, amounts, concepts)
  - All debit notes applied (with dates, amounts, concepts)
  - Current balance / net total (after all notes)
- Use this for both:
  - Manual invoice PDF download
  - Manual credit/debit note PDF download

### Key Design Decisions to Make
1. **PDF library** ‚Äî `pdfkit` (streaming, NestJS-friendly) vs `pdf-lib` (modification-friendly) vs keep raw PDF
2. **Where to persist credited items** ‚Äî Currently items are NOT stored for notes. To show them in a PDF, either:
   a. Store items in a new `credit_note_items` / `debit_note_items` table
   b. Derive from invoice items at PDF-generation time using the note's correctionConceptCode
   c. Accept less detail in the PDF
3. **Note numbering for manual notes** ‚Äî Use `NC-MAN-{invoiceNumber}-{seq}` pattern
4. **Cancellation behavior** ‚Äî For manual invoices, should a credit note with concept "2" set status to CANCELLED? Probably yes ‚Äî it's an internal state change.
