# Propuesta: invoice-numbering-emission

## Intención

Reemplazar el numbering secuencial actual basado en `COUNT+1`/`MAX+1` — que sufre race conditions en la constraint UNIQUE de `invoiceNumber` — por un auto-increment real administrado por PostgreSQL mediante `GENERATED BY DEFAULT AS IDENTITY`. Adicionalmente, crear la entidad `InvoiceElectronicEmission` para persistir todos los campos de respuesta de Factus (CUDE, QR URL, validación, totales) en lugar de descartarlos como ocurre hoy con un único `factusNumber`.

---

## Alcance

### In Scope

1. Agregar columna `sequentialNumber INT GENERATED BY DEFAULT AS IDENTITY` a la entidad `Invoice`.
2. `invoiceNumber` pasa a ser un string de display derivado: `FAC-{seq}` (electrónica) o `MAN-{seq}` (manual).
3. Nueva entidad `InvoiceElectronicEmission` con todos los campos de respuesta de Factus: number, cude, qrUrl, publicUrl, isValidated, validatedAt, emittedAt, numberingRange, items, taxes, totals, links.
4. Migrar datos existentes de `factusNumber` a la nueva entidad.
5. Eliminar columna `factusNumber` de `invoices`.
6. `downloadDianPdf()` usa `emission.number` en lugar de `factusNumber || invoiceNumber`.
7. Modelo frontend actualizado con `sequentialNumber` y `emission`.
8. Backward compatibility en display de facturas existentes.

### Out of Scope

- Auto-increment para notas crédito/débito (tienen su propia lógica de numbering).
- Cacheo local de PDFs DIAN.
- Re-intento automático de emisiones fallidas.
- Convertir facturas manuales a electrónicas post-creación.

---

## Enfoque

### 1. sequentialNumber — Auto-increment

Agregar a la entidad `Invoice`:

```typescript
@Column({
  name: 'sequential_number',
  generated: 'identity',
  type: 'int',
  generatedIdentity: 'BY DEFAULT',
})
sequentialNumber: number;
```

PostgreSQL asigna el valor automáticamente en INSERT. Al ser `BY DEFAULT`, permite backfill manual durante la migración de datos existentes.

### 2. invoiceNumber derivado

`invoiceNumber` se calcula en el servicio al crear la factura:

- Electrónica: `FAC-${sequentialNumber.toString().padStart(6, '0')}`
- Manual: `MAN-${sequentialNumber.toString().padStart(6, '0')}`

La constraint UNIQUE se mantiene. Para facturas electrónicas emitidas, `invoiceNumber` puede ser sobreescrito por el número asignado por Factus (e.g., `SETP990003678`).

### 3. InvoiceElectronicEmission

Nueva entidad con relación `@OneToOne(() => Invoice)` que almacena:

- `factusNumber` — número DIAN asignado
- `cude` — CUDE/CUFE de DIAN
- `qrUrl`, `publicUrl` — URLs de consulta
- `isValidated`, `validatedAt` — estado de validación DIAN
- `emittedAt` — timestamp de emisión
- `numberingRange` (prefix, from, to, resolution) — rango de numeración Factus
- `total`, `taxAmount` — totales financieros
- `rawResponse` — JSONB con respuesta completa para debugging

### 4. Migración de datos

Fase 1: Crear tabla `invoice_electronic_emissions` y columna `sequential_number`.

Fase 2: Migrar registros existentes con `factusNumber IS NOT NULL` a la nueva entidad.

Fase 3: Backfill `sequential_number` para filas existentes usando secuencia temporal.

Fase 4: Eliminar columna `factusNumber`.

### 5. Frontend

El modelo `Invoice` incorpora:
- `sequentialNumber: number`
- `emission?: { factusNumber, cude, qrUrl, publicUrl, isValidated, ... }`

La UI sigue mostrando `invoiceNumber` como identificador principal.

---

## Decisiones Clave

| Decisión | Opción | Razón |
|----------|--------|-------|
| Tipo de auto-increment | `GENERATED BY DEFAULT AS IDENTITY` | Semántica SQL estándar, permite backfill, evita race conditions. |
| Relación Invoice → Emission | OneToOne opcional | Una factura tiene 0 ó 1 emisión electrónica. |
| Almacenamiento raw response | `JSONB` | Permite debugging sin schema rígido para campos variables de Factus. |

---

## Riesgos

| # | Riesgo | Mitigación |
|---|--------|------------|
| R1 | Gaps en sequentialNumber por rollbacks de transacciones | Aceptado como comportamiento natural de secuencias. No afecta la unicidad ni el display. |
| R2 | Migración de factusNumber incompleta | Verificar con `COUNT(*)` antes de eliminar columna. Migración en transacción. |
| R3 | Frontend consume `factusNumber` directamente | Reemplazar referencias por `emission?.factusNumber`. Mantener getter temporal si es necesario. |
