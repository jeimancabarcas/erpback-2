# Especificación: invoice-numbering

## Propósito

Esta especificación describe el nuevo sistema de numeración auto-incremental para facturas. Se reemplaza el approach actual basado en `COUNT+1`/`MAX+1` — propenso a race conditions — por un `sequentialNumber` administrado por PostgreSQL mediante `GENERATED BY DEFAULT AS IDENTITY`. El `invoiceNumber` se convierte en un string de display derivado del `sequentialNumber`.

---

## Requisitos

### Requisito: sequentialNumber auto-incremental

La entidad `Invoice` DEBE incluir un campo `sequentialNumber` de tipo `INT` mapeado a la columna `sequential_number` con generación `GENERATED BY DEFAULT AS IDENTITY`. PostgreSQL asigna automáticamente el valor en cada INSERT.

**Escenario: Creación de factura asigna sequentialNumber único**
- DADO: no existen facturas en la base de datos
- CUANDO: se persiste una nueva factura mediante `INSERT`
- ENTONCES: la fila resultante tiene `sequential_number = 1` sin intervención del código aplicación

**Escenario: Múltiples facturas concurrentes reciben números distintos**
- DADO: dos solicitudes simultáneas de creación de factura
- CUANDO: ambas transacciones se insertan concurrentemente
- ENTONCES: cada factura recibe un `sequentialNumber` diferente garantizado por la secuencia de PostgreSQL

**Escenario: Rollback de transacción no reusa el sequentialNumber**
- DADO: una transacción fallida que insertó una factura y hizo rollback
- CUANDO: la siguiente factura se crea exitosamente
- ENTONCES: el `sequentialNumber` tiene un gap (el valor reservado por la transacción fallida no se reusa)

---

### Requisito: invoiceNumber derivado del sequentialNumber

El `invoiceNumber` DEBE generarse a partir del `sequentialNumber` usando el formato `FAC-{seq-padded}` para facturas electrónicas y `MAN-{seq-padded}` para manuales, con padding a 6 dígitos.

**Escenario: Factura electrónica recibe FAC-{sequentialNumber}**
- DADO: una factura con `isElectronic = true` y `sequentialNumber = 42`
- CUANDO: se genera el `invoiceNumber`
- ENTONCES: el valor resultante es `"FAC-000042"`

**Escenario: Factura manual recibe MAN-{sequentialNumber}**
- DADO: una factura con `isElectronic = false` y `sequentialNumber = 7`
- CUANDO: se genera el `invoiceNumber`
- ENTONCES: el valor resultante es `"MAN-000007"`

**Escenario: invoiceNumber sobreescrito por Factus en emisión electrónica**
- DADO: una factura electrónica con `invoiceNumber = "FAC-000042"`
- CUANDO: `emit()` recibe respuesta exitosa de Factus con `number = "SETP990003678"`
- ENTONCES: el `invoiceNumber` se actualiza a `"SETP990003678"` y `sequentialNumber` permanece en 42

---

### Requisito: Backward compatibility de invoiceNumber

El `invoiceNumber` DEBE seguir siendo único (constraint UNIQUE en la columna) y DEBE seguir siendo el identificador de display principal en respuestas API. Los consumidores existentes de la API NO DEBEN requerir cambios.

**Escenario: API response incluye invoiceNumber**
- DADO: una factura creada con el nuevo sistema de numeración
- CUANDO: `GET /sales/invoices/{id}` retorna la factura
- ENTONCES: el campo `invoiceNumber` está presente en la respuesta con el formato `FAC-XXXXXX` o `MAN-XXXXXX`

**Escenario: invoiceNumber mantiene unicidad**
- DADO: dos facturas con distintos `sequentialNumber` y una con `isElectronic = true` y otra con `isElectronic = false`
- CUANDO: se derivan los `invoiceNumber`
- ENTONCES: ambos valores son distintos debido al prefijo `FAC-` vs `MAN-` y cumplen la constraint UNIQUE
